# Vault2077 OPC 服务台首版设计基线

> 状态：规范性文件。产品与页面规则已冻结；正式服务内容、价格和联系方式属于上线输入。

## 1. 产品任务与边界

OPC 服务台把一人公司常见的专业工作封装成可比较、可准备、可直接交付的标准服务。用户在网站理解范围和价格，扫描统一工作人员二维码或复制账号，随后在线下完成适用性确认、付款和交付。

首版不建设账户、购物车、在线支付、订单、材料上传、进度查询、在线客服或交付物下载。网站不得使用“购买”“下单”“立即支付”等会造成站内交易预期的措辞。

频道标语为“一人公司，全栈运行”。

## 2. 不可破坏的规则

1. Vault2077 是标准服务的直接交付方，不把第三方名录包装成自身服务。
2. 每项公开服务必须拥有确定的服务范围、价格文本、材料、周期、成果和专业复核类型。
3. 工作人员必须在收款前完成适用性确认；超出服务范围的需求不能按标准服务直接成交。
4. 价格是否含税、官方费用是否另计、周期从何时开始计算必须明确。
5. 示例服务不得以生产状态发布；正式内容必须由对应专业负责人确认。
6. 页面上的价格、范围和修订号必须来自同一服务修订，禁止字段错配。
7. 二维码必须同时提供可复制的文字联系方式。

## 3. 领域记录

### 3.1 服务分类

首发分类固定为法务、税务与财务、知识产权、申报与备案、人力资源、传媒与传播。新增分类属于产品决策，必须更新总规格和数据枚举。

### 3.2 标准服务

```text
Service
  id, category_id, code, slug, name, short_audience,
  price_text, tax_note, official_fee_note,
  audience, unsuitable_for,
  inclusions, exclusions, required_materials,
  delivery_time, delivery_start_condition,
  deliverables, reviewer_type,
  revision, effective_at, status, updated_at
```

`price_text` 是对用户生效的完整价格表达，可以是固定金额或明确的“起”价；不得保存无法解释的裸数字。`delivery_time` 必须与 `delivery_start_condition` 同时出现。

### 3.3 联系渠道

```text
ContactChannel
  id, label, display_text, qr_object_key,
  availability_note, status, updated_at
```

首版全站只使用一个处于 `active` 状态的统一工作人员渠道。更换二维码与文字账号必须原子发布，避免两者指向不同工作人员。

## 4. 服务生命周期

状态固定为：

```text
draft → professional_review → ready → published → retired
                     ↘ rejected
```

- `draft`：运营人员正在录入，不可公开。
- `professional_review`：等待对应律师、税务师或专业负责人确认。
- `ready`：字段完整且已确认，等待发布。
- `published`：公开可见并允许引导联系。
- `retired`：停止接受新咨询；旧稳定 URL 显示“已停止提供”，不得变成 404。
- `rejected`：本修订不能发布，修改后形成新修订。

已经产生咨询的公开修订不得被静默改写。价格、范围、周期、成果或责任边界改变时必须提高修订号并保留旧版生效时间。

## 5. OPC 首页

页面顺序：

1. 统一频道大标语和一句话说明；
2. 状态说明：正式菜单或明确的示例状态；
3. 按分类组织的服务记录；
4. 交付方式说明；
5. 统一联系方式；
6. 常见问题；
7. 全站紧凑页脚。

每条服务记录只显示：服务编号、分类、名称、一行适用对象、价格、周期和修订号。整条记录可进入详情。首页不出现促销价、倒计时、销量、推荐标签或虚构评价。

服务较多时按分类自然分段，不使用首页搜索。分类文字可以作为页内锚点，但不得做成遮挡内容的标签区。

## 6. 服务详情

详情按以下顺序展示：

1. 服务编号、分类、修订号和生效日期；
2. 服务名称、固定价格及价格说明；
3. 适用对象和不适用情况；
4. 包含事项；
5. 不包含事项；
6. 所需材料；
7. 标准周期及起算条件；
8. 交付成果；
9. 专业复核类型；
10. 统一联系方式和“联系工作人员确认适用性”；
11. 专业服务责任边界和隐私入口。

包含与不包含事项使用平行、可核对的短句。页面不得用概括性营销文案替代范围清单。

## 7. 联系交接

主操作统一命名为“联系工作人员”。点击后可以显示二维码、复制账号和以下指引：发送服务名称或编号、简述需求、等待适用性确认。不得在公开分析事件中记录用户随后发送的内容。

如果联系渠道不可用：

- 页面保留服务内容，但隐藏二维码和失效账号；
- 主操作替换为“当前暂停接受咨询”；
- 不提供虚假的备用联系方式；
- 后台和发布监控产生阻断告警。

## 8. 内容质量与状态

- 价格不确定：服务保持 `draft`，不得显示“面议”冒充固定价格服务。
- 材料或周期依情况变化：必须在服务范围中给出确定条件，否则不属于标准服务。
- 官方办理周期不可控：分别写明 Vault2077 处理周期和官方流程周期。
- 一个需求横跨多个服务：由工作人员线下判断，不在网站自动组合套餐。
- 服务停售：稳定 URL 保留停售说明和返回服务台入口。
- 正式菜单为空：首页显示“服务范围正在由专业团队确认，暂不接受咨询。”

## 9. 视觉与响应式

OPC 延续全站纸张灰阶和文字账本，但可用 `--instrument` 深色区域表达直接交付责任。列表采用直角记录行，不使用价格卡片、徽章、图标或插画。游骑兵协会的专家头像是唯一图像例外：统一允许用于身份识别；正式档案必须与经本人确认的公开姓名、职业身份和档案状态绑定，明确标注为预览的档案可使用预览头像。头像保持灰阶、直角且不作为装饰背景。悬浮与键盘聚焦整行反色，标题轻微右移。

移动端记录顺序为名称与适用对象、价格、周期、编号与修订；价格和周期不得被折叠到详情后。二维码区域必须允许用户长按识别，并保留复制账号按钮。

## 10. 运营边界与分析

运营人员可以创建修订、送专业复核、发布、停售和更换统一联系渠道。后台不得替代律师或税务师完成专业确认，也不建设站内客户管理。

允许记录：服务列表访问、服务详情访问、联系方式展开、账号复制。核心业务指标“有效咨询”由运营人员按月依据真实沟通人工确认，不能用二维码曝光次数代替。

## 11. 验收标准

- 任一公开服务可以在三次操作内到达联系方式。
- 用户无需沟通即可明确价格、范围、材料、周期和成果。
- 所有生产服务均有专业确认、修订号和生效日期。
- 示例服务、缺失价格、缺失范围或缺失联系方式时生产发布失败。
- 二维码不可用时仍可复制文字账号；两者不可用时页面明确暂停咨询。
- 360px、768px、1280px 和 1440px 下价格与周期可读且不横向溢出。

## 12. 上线输入

- 经专业负责人确认的正式服务目录；
- 正式价格、税费与官方费用口径；
- 每项服务的范围、材料、周期和成果；
- 统一二维码、工作人员账号和可服务时间说明；
- 企业主体、发票、隐私和专业责任边界文本。

这些输入缺失不阻止页面与数据模块开发，但阻止 OPC 从示例模式切换到生产模式。
