# 所有境外公开数据统一进入一套采集与投递管线

## 背景

当前实现同时存在 Vault HMAC 内容批次、SiC Bearer 内容包、SiC 境内快照刷新和边境计划境内 GitHub 刷新。它们分别维护调度、认证、Schema、失败状态和监控，导致同一类跨区可靠性问题被重复实现，也使境内部署仍需要主动访问多个境外上游。

项目最终部署在中国大陆境内 VPS。资讯、论文、官方档案、课程、播客、模型与扩展排行榜、GitHub 趋势和边境计划所需公开仓库状态都属于境外公开数据；用户隐私、后台状态、数据库与 LLM 配置必须留在境内。

## 决策

建立一个独立、无状态的境外深模块 `vault2077-collector`。模块的外部 interface 只有：

1. 输入一版来源注册表、采集窗口和可选的公开任务；
2. 输出不可变 `AcquisitionBatch` 与完整 `sourceReports`；
3. 使用统一 HMAC 协议把批次投递到境内 `POST /api/internal/acquisition`。

资讯、SiC 固定内容、GitHub/Hugging Face/OpenRouter/Skill/MCP 排行状态和边境公开仓库检查都作为模块内部 adapter。adapter 可以使用不同语言或依赖，但不得把自己的认证、队列、调度、投递格式或状态接口暴露给工作流。

GitHub Actions 每小时固定触发同一个工作流。来源可以在注册表声明整数小时 cadence，但所有到期判断、采集、签名、投递、artifact 保存和告警都属于同一运行。强制赛季快照和交互式公开仓库验证仍复用同一模块与批次协议，不另建境内直连 GitHub 的生产路径。

境内接收端先验签和不可变持久化，再异步处理。翻译、摘要、内容分类、事件编排和中文项目说明只使用境内 LLM；原始数值排名由确定性代码计算，LLM 不得改变观测值或排序。只有进入已验证公开版本的处理结果才能出现在网站，上一成功版本在任何采集或模型失败时继续可读。

每次运行使用稳定 `runId`，每个批次使用 `batchId`，每条记录使用按来源身份和内容哈希生成的幂等 ID。GitHub Actions、境内接收、队列、LLM、隔离/死信和公开版本必须能通过这些标识串联。

## 安全边界

- 境外只处理公开数据，不接收邮箱、后台会话、咨询记录或数据库内容。
- 境外只持有只读上游凭据和可轮换投递密钥。
- 每一跳重定向前校验协议、获批主机和解析后 IP，阻止私网、环回、链路本地和保留地址。
- 所有原文、Feed、README、HTML 和模型输入均视为不可信数据。
- 批次设置记录数、单记录字节数、总字节数、压缩比、时间窗和 Schema 版本上限。

## 迁移

现有 Vault `ContentBatch`、SiC `SicRawCollection`、SiC snapshot 和 Frontier tick 暂时作为兼容 adapter 保留。迁移顺序为：

1. 冻结统一 Schema、状态机和来源注册表；
2. 建立统一境外 CLI、HMAC 投递和境内接收；
3. 依次迁移 Vault、SiC 固定内容、各排行榜、边境公开仓库状态；
4. 完成双写比对、故障注入和七天连续运行；
5. 删除旧 Bearer、境内主动抓取和独立 cron。

## 结果

调用方只需要理解一套采集 interface，跨区可靠性、安全和可观测性集中维护；代价是迁移期需要兼容旧批次并建立多记录类型 Schema。该代价由更高的可审计性、故障定位能力和长期稳定性抵消。
