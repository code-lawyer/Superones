# Vault2077 边境计划网站 MVP 设计基线

> 状态：规范性文件。更新时间：2026-07-22。本文冻结边境计划网站 MVP 的目标、理念、页面、资格、排名、奖品捐献和自动结算规则；站外联系、奖品交付与其他频道不在本文范围内。

## 1. MVP 目标

边境计划网站 MVP 只证明一件事：一套没有用户账户和主观评审的公开建造机制可以完整运行一个真实赛季。

完整闭环为：

```text
公开仓库报名
  → 挑战文件验证
  → 记录 Star 基线
  → 每小时更新实时排名
  → 季度结算时重新验证资格
  → 冻结最终排名与季度冠军
  → 按排名顺序随机分配匿名奖池
  → 公布本季结果并进入往届记录
```

MVP 是否成功，以至少完整跑完一个真实季度闭环为准，不以流量、报名数、捐献数量或商业收入为验收条件。

## 2. 核心理念

### 2.1 固定大标语

边境计划页面必须遵循 Vault2077 全站设计规范。现有 `PageIntro` 的大标语固定为：

> 无垠荒野，永不落幕

不得为了边境计划单独更换全站字体、色彩、栅格、导航或交互语言，也不得把页面改造成常见活动官网、彩色赛事页或赞助商展示页。

### 2.2 四无

大标语之后设置独立的全宽“四无”区块，固定顺序为：

> 无期限 · 无评审 · 无组织 · 无目标

- **无期限**：计划全年持续运行，任何时间均可报名，每个自然季度结算一次。
- **无评审**：没有评委、赛道、主观评分或项目质量判断；排名只由报名后的净新增 Star 决定。
- **无组织**：网站执行公开、机械、可复算的规则，不以主办方、赞助商或主理人的意志改变名次。
- **无目标**：不规定项目类型、技术栈、完成度、商业目的或创作方向。

“四无”不等同于没有机器可验证的最低资格。公开仓库、非纯 Fork、未归档、开源许可证和仓库控制权验证仍然是报名条件。

### 2.3 宣言

大标语下方提供“查看边境计划宣言”按钮，点击后以可访问弹窗展示本文附录 A 的完整宣言。

宣言不替代参赛规则。宣言解释计划为何存在；参赛规则只定义报名、验证、排名、结算和隐私等操作性条件。

### 2.4 统一称谓

公开文案称报名者为“穿越者”，不称“选手”“用户”“团队”或“参赛账号”。系统仍以仓库作为报名、排名和结果的唯一公开主体。

## 3. 网站范围

### 3.1 MVP 包含

1. 边境计划首页；
2. 项目报名与挑战文件验证；
3. 当前赛季完整排行榜；
4. 官方冠军奖励和匿名随机奖池；
5. 奖品捐献页面；
6. 报名规则弹窗；
7. 宣言弹窗；
8. 首页底部的往届记录；
9. 现有共享密码后台中的最小奖品确认界面；
10. 每小时 Star 更新、季度结算和一次性随机分配。

### 3.2 MVP 不包含

- Vault2077 普通用户账户；
- 报名者或捐献者个人主页；
- 支付、众筹、物流、仓储或所有权托管；
- 站内消息、站内选奖或客服工作流；
- 评论、投票、赛道、评委或项目审核；
- 赞助商展示、推广链接或捐献者署名；
- 独立规则页和独立历史页；
- 复杂奖品库存与履约系统。

站外联系和奖品实际交付不属于网站功能。网站只保存必要联系邮箱、公开奖品状态和最终分配结果。

## 4. 信息架构与页面设计

### 4.1 边境计划首页 `/frontier`

页面自上而下固定为：

1. **全站 PageIntro**：保留“大标语 + 说明 + 当前赛季元数据”的既有结构，大标语不得修改；在开幕说明下提供“查看边境计划宣言”。
2. **四无宣言带**：全宽展示四个固定原则，每项只配一句解释。
3. **当前赛季**：赛季名称、报名状态、最近更新时间、结算时间和主操作“参与计划”；旁边提供“查看参赛规则”弹窗入口。
4. **实时排行榜**：首页展示前五名并链接到完整榜单。
5. **奖品区**：先展示官方冠军奖励，再展示匿名随机奖池；“捐献奖品”位于随机奖池标题区，操作层级低于“参与计划”。
6. **往届记录**：紧凑展示历季冠军、最终净新增 Star 和随机奖品分配结果。

首页删除现有“谁可以参加”长规则区块。资格与隐私条款只在规则弹窗和报名流程中完整呈现。

### 4.2 报名页 `/frontier/submit`

报名页只收集：

- GitHub 仓库地址；
- 联系 Email；
- 一句话项目说明。

页面提供“查看完整规则”按钮。提交前必须勾选“我已阅读并同意边境计划参赛规则”；未勾选时不得生成挑战文件。

报名流程继续使用现有三步结构：

```text
填写报名 → 创建并提交挑战文件 → 验证成功
```

### 4.3 完整排行榜 `/frontier/ranking`

每行展示：

- 排名；
- 仓库和一句话项目说明；
- Star 基线；
- 最近一次成功抓取的当前 Star；
- 有符号净新增 Star；
- 验证日期。

榜单顶部必须显示最近更新时间。页面不得使用“实时计数器”制造秒级实时错觉。

### 4.4 奖品捐献页 `/frontier/donate`

表单只收集：

- 奖品名称；
- 奖品说明；
- 联系 Email。

《奖品捐献须知》直接显示在表单上方或提交按钮之前，不放入弹窗。提交前必须勾选“我已阅读并同意奖品捐献须知”。Email 只做格式校验，不发送验证码。

页面明确显示本次捐献预计进入的赛季。提交成功后只返回“已提交，等待确认”，不提供捐献者账户或公开查询入口。

### 4.5 弹窗行为

宣言弹窗和规则弹窗必须：

- 使用全站既有灰阶、边线和排版；
- 打开后把焦点移入弹窗；
- 支持关闭按钮和 `Escape`；
- 关闭后把焦点还给触发按钮；
- 移动端允许正文独立滚动；
- 不使用独立页面替代弹窗。

## 5. 赛季

边境计划全年持续运行，赛季按北京时间自然季度自动生成：

| 月份 | 显示名称 | 结算时间 |
| --- | --- | --- |
| 1–3 月 | 春季赛 | 3 月 31 日 23:59:59 |
| 4–6 月 | 夏季赛 | 6 月 30 日 23:59:59 |
| 7–9 月 | 秋季赛 | 9 月 30 日 23:59:59 |
| 10–12 月 | 冬季赛 | 12 月 31 日 23:59:59 |

当前日期 2026-07-22 对应“2026 秋季赛”。赛季名称和结算时间必须由日期规则生成，不得在页面和存储模块分别硬编码。

状态固定为：

```text
open → settling → settled → archived
```

- `open`：全赛季接受报名、挑战验证和奖品捐献。
- `settling`：停止新报名和验证，执行最终资格检查、最终 Star 抓取和随机分配。
- `settled`：结果已冻结并公开，新赛季可以开始。
- `archived`：结果保留在往届记录中。

一次报名只参加验证成功后最近一次季度结算，不自动续入下一赛季。非冠军仓库可以在后续赛季重新报名，并记录新的 Star 基线。

## 6. 报名资格

网站只执行以下机器可验证条件：

1. GitHub 仓库公开；
2. 不是纯 Fork；
3. 未归档；
4. 具有 GitHub 可识别的开源许可证；
5. 默认分支可读取；
6. 同一仓库在同一赛季没有其他已验证报名；
7. 仓库未进入季度冠军名册；
8. 挑战文件能够证明报名者可以修改仓库。

网站不得根据项目类型、技术栈、完成度、README 质量、项目价值、商业属性、提交数量或主动传播行为进行主观判断。不得使用“空壳”“垃圾项目”“搬运”“刷 Star”等无法由上述机器规则精确定义的理由拒绝报名。

同一 Email 可以为多个不同仓库报名。联系 Email 永不出现在公开页面或接口中。

## 7. 挑战文件

### 7.1 生成与初次验证

系统为“赛季 + 标准化仓库”生成高熵挑战码，并要求穿越者在默认分支创建：

```text
.vault2077/season-{season-code}.json
```

文件内容至少包含：

```json
{
  "platform": "vault2077",
  "season": "2026-Q3",
  "repository": "owner/repository",
  "challenge": "一次性随机字符串",
  "issuedAt": "2026-07-22T12:00:00+08:00"
}
```

待验证挑战 24 小时内有效，只能用于对应仓库和赛季。验证成功后立即记录 Star 基线，并保留挑战哈希至赛季结算完成。

### 7.2 保留与结算复查

挑战文件必须保留在默认分支，内容不得修改，直至赛季结算完成。

网站只在两个时间点读取挑战文件：

1. 报名验证时；
2. 赛季结算时。

网站不在每小时排行榜更新时检查挑战文件。结算时文件缺失或内容不匹配的仓库失去本赛季最终排名、季度冠军和随机奖品资格。

报名页面和规则弹窗必须醒目标明这条保留义务。

## 8. 排名与 Star 更新

### 8.1 基线与公式

挑战验证成功时只写入一次 Star 基线：

```text
net_new_stars = current_stars - baseline_stars
```

净新增 Star 使用有符号整数；取消 Star 时允许显示负数，不得截断为零。

### 8.2 更新频率

开放赛季每小时读取一次所有已验证仓库的 GitHub Star，并保存抓取时间。新报名验证成功后立即进入榜单。

普通页面访问只读取最近一次成功结果，不直接触发 GitHub 请求，也不提供用户手动刷新按钮。

### 8.3 排序

排序依次为：

1. 净新增 Star 较高；
2. 当前累计 Star 较高；
3. 验证通过时间较早；
4. 标准化仓库名升序。

排序规则自动执行，人工不得修改 Star、基线、净新增值或名次。

## 9. 奖励体系

### 9.1 官方冠军奖励

每个赛季只有一个季度冠军。季度冠军是最终资格有效且净新增 Star 排名第一的仓库。

官方冠军奖励：

- 由边境计划提供；
- 在页面上与随机奖池分开显示；
- 不进入随机抽取；
- 只授予季度冠军。

季度冠军仍然参加该赛季随机奖池分配。同一冠军仓库永久退出后续赛季；获得随机奖品但未成为冠军的仓库仍可重新报名。

### 9.2 匿名随机奖池

随机奖池只由已确认的匿名奖品捐献组成。公开奖品记录只显示名称、说明、适用赛季和状态，不显示捐献者身份、机构、头像、宣传链接或 Email。

一条捐献记录代表一件完整奖品和一次随机分配。网站不解析说明中的数量，也不把一条记录拆分给多个仓库。

## 10. 奖品捐献

### 10.1 提交与状态

奖品捐献状态固定为：

```text
pending_confirmation → available → assigned
          │                ├──────→ carried_over
          └────→ rejected  └──────→ withdrawn
```

- `pending_confirmation`：已提交，尚未进入公开奖池。
- `available`：已确认并显示在指定赛季随机奖池中。
- `rejected`：未通过确认，不公开。
- `withdrawn`：确认后、结算前撤回，不参与分配。
- `assigned`：已在结算时随机分配给一个仓库。
- `carried_over`：本季未被分配，顺延到下一赛季。

奖品提交时默认关联当前赛季。只有在结算前进入 `available` 的奖品才参加本季分配；结算时仍为 `pending_confirmation` 的奖品自动转入下一赛季等待确认。

### 10.2 最小内部确认界面

复用现有共享密码后台，仅增加“边境计划 / 奖品捐献”列表，支持：

- 查看待确认奖品的名称、说明和 Email；
- 确认并加入奖池；
- 拒绝；
- 在结算前撤回已确认奖品。

后台不提供修改奖品文案、调整排名、手动指定随机结果或重复抽取的能力。

### 10.3 奖品捐献须知

捐献页面使用以下 MVP 文案基线：

> 提交即表示你自愿将所述奖品加入边境计划匿名随机奖池，并确认奖品真实存在、说明真实准确。奖品被分配后，你同意无偿向获得该奖品的穿越者转移奖品所有权或相应使用权，并通过提交的 Email 完成必要联系与交付。你的身份和联系方式不会在公开奖池中展示。奖品提交后须经确认才会公开；在随机分配完成前可以申请撤回，分配完成后不得以宣传、付费或其他附加条件拒绝交付。

表单必须把须知与具体版本号一同记录，并保存勾选同意时间。

## 11. 季度结算与随机分配

### 11.1 最终资格

赛季进入 `settling` 后停止新报名和验证。网站重新检查每个已验证仓库：

- 仍为公开仓库；
- 仍非纯 Fork；
- 仍未归档；
- 仍有可识别开源许可证；
- 默认分支仍可读；
- 挑战文件仍存在且内容与报名记录完全匹配。

任一条件不满足时，该仓库不进入最终排名、冠军判定或随机奖池分配。实时榜单可以保留赛季过程数据，但最终结果必须明确显示其资格失效。

### 11.2 锁榜

所有有效仓库取得结算时间点的最终 Star 后，网站使用固定排序规则冻结最终排名。最终快照未完成时不得继续冠军判定和随机分配。

### 11.3 随机分配算法

随机分配只执行一次：

1. 读取最终有效排名，从第一名开始依次处理；
2. 对当前仍可用的奖品使用服务端加密安全随机数选择一件；
3. 保存“仓库 + 奖品 + 顺序 + 分配时间”；
4. 将该奖品从可用集合移除；
5. 每个仓库最多获得一件随机奖品；
6. 参赛仓库处理完毕或奖池清空时停止；
7. 剩余奖品进入 `carried_over`，自动关联下一赛季。

季度冠军先获得官方冠军奖励，并照常参与上述随机分配。

每个赛季只能存在一份随机分配结果。结果一旦保存，不提供重新抽取、人工换奖或站内选奖。

### 11.4 公开结果

结算完成后公开：

- 最终有效排名和最终快照时间；
- 季度冠军与官方冠军奖励；
- 每个获得随机奖品的仓库及对应奖品；
- 随机分配执行时间；
- 资格失效仓库的公开状态；
- 顺延到下一赛季的剩余奖品。

结果进入首页往届记录。新赛季按自然季度自动开始，不等待站外联系或奖品交付完成。

## 12. 状态与失败反馈

### 12.1 报名

- 仓库不符合机器资格：明确指出具体失败条件。
- 挑战过期：允许重新生成，旧待验证记录失效。
- GitHub 暂时不可用：保留当前步骤，允许稍后重试，不创建重复报名。
- 同仓库同赛季已验证：返回现有报名状态，不生成第二条记录。
- 冠军仓库再次报名：明确说明该仓库已进入往届冠军记录。

### 12.2 排行榜

- 本赛季无报名：显示“本赛季尚无通过验证的项目。提交第一个公开仓库。”
- 每小时抓取失败：保留上一成功值并显示真实更新时间。
- 净新增为负：显示负数，不使用 `+0` 掩盖。

### 12.3 奖品捐献

- 未同意须知：禁止提交。
- Email 格式错误：在字段附近显示错误。
- 提交成功：显示“奖品已提交，确认后将进入对应赛季随机奖池。”
- 本季已进入结算：新提交自动指向下一赛季。

### 12.4 结算

- 最终 GitHub 数据或挑战文件检查尚未完成：保持 `settling`，不得发布部分随机结果。
- 随机分配已经存在：重复任务直接返回既有结果，不得再次抽取。
- 没有可用奖品：仍然正常确定季度冠军并发布最终排名。

## 13. 隐私与公开边界

公开字段：仓库、项目说明、验证日期、基线 Star、当前 Star、净新增 Star、最终资格、冠军结果和随机奖品分配。

私密字段：报名 Email、奖品捐献 Email、挑战哈希、规则同意记录、捐献须知同意记录和内部确认状态说明。

- Email 只用于对应报名或奖品捐献的必要联系，不公开展示。
- 网站不接受用户提供 GitHub Token。
- 挑战码不是密钥或授权，不得在文案中称为 Token。
- 宣言与四无理念不得被解释为允许绕过网站的机器资格、隐私边界或随机分配结果。

## 14. 验收标准

### 14.1 页面与设计

- 大标语“无垠荒野，永不落幕”保持不变。
- 四无以独立全宽区块展示，符合全站灰阶、字体、边线和响应式规则。

### 14.2 报名与排名

- 无 Vault2077 账号即可完成报名和挑战验证。
- 规则未勾选时不能生成挑战。
- 挑战不能跨仓库、跨赛季、重复使用或在初次验证过期后使用。

- 一人可报名多个不同仓库，同仓库同赛季不能重复验证。
- Star 基线只写入一次；榜单每小时更新并显示时间。
- 负增长正确显示；排序可由公开值复算。

### 14.3 奖品与结算

- 捐献表单只有奖品名称、说明和 Email，并强制同意须知。
- 未确认奖品不出现在公开奖池；公开奖池不显示捐献者身份或 Email。

- 只有季度冠军仓库永久退出；随机奖品获得仓库可以参加后续赛季。
