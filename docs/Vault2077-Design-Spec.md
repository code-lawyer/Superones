# Vault2077 产品与设计规格

> 状态：已确认，可进入第一阶段开发  
> 版本：1.0  
> 日期：2026-07-21  
> 产品主张：**一人公司，全栈运行。**

## 1. 文档目的

本文档定义 Vault2077 首发版本的产品边界、信息架构、页面体验、内容模型、视觉系统、数据与系统架构、风控要求、开发阶段和验收标准。它是设计与工程实现的共同基线。

首发开发按三个阶段推进：

1. 高保真公开站：完成品牌首页、四个频道、响应式页面、真实路由和示例数据。
2. 平台能力：接入数据库、轻量运营后台、GitHub 仓库验证与排行榜。
3. 自动化内容：部署境外采集、境内 LLM 翻译摘要、事件聚合与自动发布管线。

### 当前实现状态（2026-07-21）

- 第一阶段公开站已完成。
- MVP 平台闭环已完成：边境计划支持无账号报名、服务端挑战码、GitHub 默认分支验证、Star 基线、公开排行榜与共享密码运营台。
- 在尚未提供大陆 PostgreSQL 实例前，MVP 将报名数据保存在部署目录的加密文件存储中。它适合单实例试运行；进入多人运营或多实例部署前，必须迁移至本规格定义的 PostgreSQL。
- 自动采集、境内 LLM 翻译摘要、来源白名单后台和真实 OPC 联系方式仍待下一阶段接入。

## 2. 产品定义

### 2.1 一句话定位

Vault2077 是面向技术型超级个体的一人公司操作系统，以持续情报为入口，以标准化专业服务为商业核心，以硅碳共生的技术趋势和永续式开放竞赛构建长期社区。

### 2.2 首发用户

核心用户是已经产生收入或即将注册经营主体的技术型超级个体，通常经营 AI 产品、开发者工具、内容/IP 或专业服务，团队规模为 1–3 人。

纯学习者、传统小微企业和普通自由职业者可以使用公开内容，但不主导首发产品设计。

### 2.3 产品层级

| 层级 | 频道 | 核心任务 | 对业务的作用 |
| --- | --- | --- | --- |
| 入口 | Vault 信息流 | 压缩高价值 AI 信息 | 获客与持续回访 |
| 商业核心 | OPC 服务台 | 展示可直接购买的标准服务 | 产生有效咨询与收入 |
| 能力层 | SiC 学院 | 发现正在加速的技术能力 | 建立专业认知与工具心智 |
| 社区层 | 边境计划 | 让公开项目持续参与竞赛 | 社区增长与品牌事件 |

### 2.4 首发商业边界

- Vault 信息流、SiC 学院和边境计划免费、无广告、无会员墙。
- OPC 服务台是网站唯一直接商业入口。
- 普通用户全站无需注册或登录。
- OPC 首发只做服务展示：用户扫码添加工作人员，在线下付款并接受服务。
- 内部运营后台使用共享密码，不建设账号、角色或组织权限系统。
- 核心成功指标为“每月由网站带来的有效 OPC 咨询数”。

### 2.5 明确不做

- 不做面向普通用户的账户、收藏、评论、私信或社区关系链。
- 不做在线支付、订单、材料上传、服务进度或交付物系统。
- 不做传统视频课程、训练营或重型学习管理系统。
- 不做开放网络的无边界抓取。
- 不做资讯全文搬运或付费内容绕过。
- 不做赞助置顶、付费榜单或人工修改竞赛排名。
- 不做图标、插画、照片、装饰性图像、3D 或粒子视觉。

## 3. 品牌与语言

### 3.1 品牌表达

主品牌：`VAULT2077`

主标语：

> 一人公司，全栈运行。

品牌说明：

> 为超级个体提供持续情报、标准化经营服务、技术趋势与开放实验场。

品牌气质应同时满足：冷静、可靠、前沿、克制、可执行。它不是游戏世界观，也不是泛 AI 媒体。

### 3.2 SiC 的定义

`Si` 代表硅，`C` 代表碳。SiC 学院代表对硅碳共生的超级个体发展道路的持续探索。

频道标语：

> 硅提供杠杆，碳决定方向。

频道说明：

> 在机器智能与人的判断之间，探索超级个体持续进化的道路。

### 3.3 界面语言

- 界面仅提供简体中文。
- 境外内容展示中文译名，并保留原始标题、原始语言、原文链接和原始发布时间。
- 不建设独立英文站。
- 所有 AI 翻译与摘要使用明确但克制的“AI 摘要”标记。

## 4. 设计原则

### 4.1 参考边界

视觉节奏参考 [Xiaomi MiMo](https://mimo.xiaomi.com/) 的单一品牌叙事、巨型文字首屏、克制留白和清晰分区，但不复制其内容结构、品牌元素或具体动效。Vault2077 的频道更复杂、数据更密集，因此首页只借鉴其“先建立主张，再展开内容”的叙事方式；频道页使用适合持续更新记录的高密度列表。

### 4.2 未来档案库，而非赛博朋克海报

视觉由文字、时间、来源、编号、状态和秩序构成。所谓“2077”体现在系统像一套来自未来的可靠基础设施，而不是霓虹城市、机器人、锁孔或终端装饰。

### 4.3 文字账本

全站的标志性设计语言是一套真实、持续更新的文字账本：

- 内容拥有稳定记录号，例如 `VLT/EVT/2026/00731`。
- 项目拥有趋势观察窗口，例如 `24H +184 / 7D +922`。
- 赛事拥有明确状态，例如 `SUMMER 2026 / OPEN / 71 DAYS LEFT`。
- 服务拥有固定范围版本，例如 `OPC/TAX/001 / REV.03`。
- 页面顶部状态行展示真实的更新时间、来源数量或赛季状态，不使用伪数据装饰。

这套账本既承担导航和可信度，也构成 Vault2077 的独特视觉签名。

### 4.4 信息压缩优先

首页不做资讯瀑布；事件不按来源机械复制；卡片不重复标题、摘要和标签。每个界面元素只完成一个任务。

### 4.5 透明而非假装客观

AI 摘要必须标识，排名方法必须解释，原始来源必须可达，事件更新时间必须可见，竞赛规则必须可复核。

### 4.6 极简必须精确

没有图像后，字号、行宽、基线、间距、分隔线和交互反馈必须严格统一。不能用大面积空白掩盖信息结构，也不能用通用卡片网格替代编辑判断。

## 7. Vault 信息流

### 7.1 来源范围

来源以白名单为主，包括：

- AI 公司官方公告、官方博客和研究页面。
- 核心人物公开账号、访谈与文章。
- 指定播客 RSS 与节目页。
- GitHub 官方发布、Release 和高价值仓库动态。
- 权威媒体和研究机构。

系统可以自动发现新来源，但新来源只进入候选池，经过一次人工批准后才能参与自动发布。

### 7.2 信息模型

信息流的核心单位是“事件”，不是“文章”。一个事件可以关联多条原始内容，并随新来源持续更新。

事件列表字段：

- 记录号
- 中文标题
- 事件类型
- 核心实体：公司、人物、产品、项目
- 一句话摘要
- 来源数量
- 首次发布时间
- 最后更新时间
- AI 摘要标记

事件详情结构：

1. 中文标题与原始标题。
2. 30–80 字结论摘要。
3. 关键事实，使用短段落而非装饰性列表。
4. 为什么值得关注。
5. 按时间排序的事件进展。
6. 全部原始来源及其发布时间。
7. 生成时间、更新时间和纠错入口。

### 7.3 列表与筛选

默认视图为“重要更新”，同时提供：最新、公司、人物、产品、播客、公告、文章。

筛选器使用文字标签，不使用下拉图标。移动端允许横向滚动文字标签，但不得隐藏当前选择。

### 7.4 自动发布管线

```text
境外白名单采集
    ↓
正文与公开元数据结构化
    ↓
安全清洗、长度限制、内容指纹
    ↓
加密传输至境内原始内容存储
    ↓
精确去重与实体提取
    ↓
时间窗口 + 向量相似度产生候选事件
    ↓
LLM 判断：同一事件 / 相关事件 / 无关
    ↓
翻译、摘要、分类与重要性评分
    ↓
高置信度自动发布或更新事件
    ↓
低置信度保持独立；失败进入运营后台
```

合并策略必须“宁可漏合并，不可误合并”。模型只能在系统提供的少量候选之间判断，不能扫描全部历史内容后自由选择事件。

### 7.5 可信度与纠错

- 每条内容标记“AI 摘要”。
- 必须保留原始来源和原文链接。
- 涉及政策、法务、税务的内容提示以官方原文和专业意见为准。
- 用户可提交纠错原因和联系方式；首发可通过邮件或轻量表单承接。
- 运营后台支持下架、重算摘要、拆分误合并事件和屏蔽来源。

## 8. OPC 服务台

### 8.1 交付模式

Vault2077 是服务的直接交付方。内部律师、税务师和其他专业人员将基础服务封装成范围明确、固定价格、固定材料清单、固定交付周期和固定成果的服务项目。

首发网站不收款、不建订单。用户浏览服务后扫描统一工作人员二维码，由工作人员在线下确认适用性、收款并交付。

### 8.2 服务分类

- 法务
- 税务与财务
- 知识产权
- 申报与备案
- 人力资源
- 传媒与传播

### 8.3 标准服务模板

每个正式服务必须具备：

| 字段 | 说明 |
| --- | --- |
| 服务名称 | 使用用户能识别的结果命名 |
| 服务编号 | 例如 `OPC/LEGAL/001` |
| 固定价格 | 明确币种及是否含税 |
| 适用对象 | 哪类一人公司适合购买 |
| 包含事项 | 本价格覆盖的工作范围 |
| 不包含事项 | 触发另行报价的边界 |
| 所需材料 | 用户开始前需要准备的内容 |
| 标准周期 | 从材料齐备后的预计工作日 |
| 交付成果 | 用户最终获得什么 |
| 专业复核 | 律师、税务师等负责人类型 |
| 版本与日期 | 服务范围最后更新时间 |

### 8.4 首发占位服务

以下内容只用于第一阶段界面与信息密度验证，正式上线前必须由专业人员替换并确认：

| 分类 | 示例服务 | 示例价格 | 示例周期 |
| --- | --- | ---: | --- |
| 法务 | 单份标准商业合同审阅 | ¥699 | 3 个工作日 |
| 法务 | 网站用户协议与隐私政策基础包 | ¥1,999 | 7 个工作日 |
| 税务 | 个体经营主体税务健康检查 | ¥899 | 5 个工作日 |
| 税务 | 一人公司基础财税咨询包 | ¥599 | 60 分钟 |
| 知产 | 单类别商标注册申请 | ¥799 起 | 以官方流程为准 |
| 知产 | 软件著作权材料整理与申报 | ¥1,499 | 以官方流程为准 |
| 备案 | ICP 备案材料预审 | ¥499 | 3 个工作日 |
| 人力 | 首位员工入职文件包 | ¥999 | 5 个工作日 |
| 传播 | 创始人基础媒体资料包 | ¥1,999 | 7 个工作日 |

这些价格不得被视为最终承诺。开发环境与设计评审页面需显示“示例内容”状态；生产发布检查必须阻止带有该状态的服务上线。

### 8.5 页面结构

- OPC 首页：服务承诺、分类索引、服务表格、交付方式、统一二维码区、常见问题。
- 服务详情：固定价格、适用/不适用、包含/不包含、材料、周期、成果、二维码操作。
- 二维码旁必须同时提供可复制的工作人员账号文字，避免桌面与无摄像头场景受阻。
- 二维码是业务材料，不属于装饰性图像；未提供正式二维码前使用纯文字占位，不生成虚假二维码。

## 9. SiC 学院

### 9.1 首发定位

SiC 学院首发不是课程平台，而是面向超级个体的技术趋势雷达。它展示趋势与自动生成的中文项目说明，不制作教程、课程或学习路径。

### 9.2 数据范围

- GitHub 公开仓库。
- 与 AI 工具、Skill、MCP、Agent、自动化和超级个体生产力相关的项目。
- 境外节点可将仓库元数据和 README 快照传入境内，供 LLM 分类和摘要。

README 始终被视为不可信输入。系统必须剥离脚本和隐藏内容，不执行其中任何指令、链接、代码或安装步骤。

### 9.3 趋势视图

- 正在爆发：以 24 小时 Star 增长为主。
- 本周热门：以 7 天 Star 增长为主。
- 长期热门：累计 Star 作为主要参考，但仍展示近期变化。

初始趋势分可采用可解释权重，后续根据真实数据调整：

```text
trend_score = 0.55 × log(1 + delta_24h)
            + 0.35 × log(1 + delta_7d / 7)
            + 0.10 × freshness_factor
```

绝对 Star 数只用于辅助展示，不能长期压制新项目。

### 9.4 项目列表字段

- 当前排名与上次排名变化
- 仓库名称
- 中文一句话说明
- 分类
- 主要语言
- 24 小时、7 天与累计 Star
- 最近更新时间
- README 抓取时间

排名变化使用 `上升 3`、`下降 2`、`新入榜` 等文字，不使用箭头图标。

### 9.5 项目详情

- 项目名称和原始仓库地址
- 中文摘要与原始描述
- 适合解决的问题
- 关键趋势数据
- README 摘要
- 许可证、主要语言、最后提交时间
- AI 摘要和抓取时间说明

首发不复制完整 README 到公开页面。

## 10. 边境计划

### 10.1 核心规则

- 全程无需用户账号。
- 参赛项目必须是公开 GitHub 仓库。
- 仓库必须包含明确的开源许可证、可运行代码或可验证成果以及用途明确的 README。
- 纯 Fork、空壳仓库、搬运项目、恶意软件和以刷 Star 为目的的页面不得参赛。
- 全赛季开放报名，验证通过时记录 Star 基线。
- 赛季结束按净新增 Star 排名。
- 每季前三名依次从当季奖池选择一件奖品。
- 未获奖仓库可在后续赛季重新报名；获得过前三名的仓库永久退出后续竞赛并进入荣誉榜。

### 10.2 赛季周期

赛季按自然季度连续运行：

| 赛季 | 结算日 |
| --- | --- |
| 春季 | 3 月 31 日 |
| 夏季 | 6 月 30 日 |
| 秋季 | 9 月 30 日 |
| 冬季 | 12 月 31 日 |

结算时间统一为北京时间 23:59:59。新赛季在下一自然日 00:00:00 开始。

### 10.3 无账号报名流程

```text
输入 GitHub 仓库地址与私人联系邮箱
    ↓
检查仓库公开性、许可证和历史获奖状态
    ↓
生成与“仓库 + 赛季”绑定的一次性挑战码
    ↓
用户提交 .vault2077/season-YYYY-QN.json
    ↓
用户返回页面点击“验证仓库”
    ↓
系统读取默认分支文件并匹配挑战码
    ↓
自动检查通过或进入人工复核
    ↓
记录当时 Star 基线并加入排行榜
```

验证文件建议格式：

```json
{
  "platform": "vault2077",
  "season": "2026-Q3",
  "repository": "owner/repository",
  "challenge": "一次性随机字符串",
  "issuedAt": "2026-07-21T12:00:00+08:00"
}
```

挑战码不是用户密钥，不具备其他权限。它只能验证提交者可以修改指定仓库的默认分支。

### 10.4 报名字段

公开字段：仓库地址、项目名称、项目说明、许可证、报名时间、基线 Star、当前 Star、净新增 Star。

私密字段：联系邮箱、挑战码、验证日志、异常复核记录。私密字段只存储在中国大陆服务端，不出现在仓库或排行榜。

### 10.5 排名与结算

```text
net_new_stars = final_star_count - baseline_star_count
```

- 排行榜展示当前 Star、基线 Star 和净新增 Star，保证结果可理解。
- 系统至少每日记录一次快照，结算日增加最终快照。
- 获奖前检查异常 Star 增长、仓库有效性、许可证、恶意内容和项目归属。
- 复核只判断资格与异常，不引入主观项目评分。
- 默认同分规则：累计 Star 较高者优先；仍相同则报名验证时间较早者优先。该规则在首季开始前写入正式赛事条款。

### 10.6 奖池

- 奖池以文字清单展示奖品名称、数量、提供方和当前状态。
- 前三名按第一、第二、第三名顺序选择尚未被选走的一件奖品。
- 奖品选择和发放由运营人员线下完成，后台只记录选择结果。
- 首发不建设捐赠支付、库存物流或自动发奖系统。

### 10.7 主要页面

- 边境计划首页：理念、当前赛季、规则、奖池、实时前三、提交入口。
- 提交页：仓库地址、邮箱、项目说明和隐私提示。
- 验证页：挑战文件内容、文件路径、提交步骤、验证状态。
- 排行榜：排名、项目、基线、当前值、净新增值、报名日期。
- 历史与荣誉榜：历季前三、奖品选择和永久退出竞赛的仓库。

## 11. 内部运营后台

### 11.1 登录

- 单一共享高强度密码。
- 密码只保存 Argon2id 哈希。
- 登录接口限速，连续失败触发短期锁定。
- 使用安全、HttpOnly、SameSite Cookie 保存短期会话。
- 后台路径不依赖“隐藏 URL”作为安全措施。
- 记录操作时间、动作和对象，但不承诺识别具体操作者。

### 11.2 功能范围

- 来源白名单：新增、暂停、恢复、查看最近采集状态。
- 内容异常：下架、恢复、重算摘要、拆分事件、屏蔽原始条目。
- 采集任务：最近运行、失败原因、重试。
- OPC 菜单：服务项目、分类、价格、状态和版本。
- 边境计划：报名复核、取消资格、赛季、奖品和结算状态。
- 系统：共享密码轮换、基础统计和操作日志。

不做富文本页面搭建器、多人审批、角色权限、复杂数据大屏。

### 12.4 网格与间距

- 桌面内容最大宽度：1440px；页面左右安全边距：32–64px。
- 采用 12 列网格，但阅读正文限制在 720–820px。
- 平板切换为 8 列，移动端为 4 列。
- 间距基准为 4px，主要区块使用 48、72、96、144px 节奏。
- 列表优先采用整行记录形式，避免把所有内容装进圆角卡片。
- 默认圆角为 0；表单控件最多 2px，不能使用大圆角胶囊。

### 12.5 分隔与编号

- 1px 分隔线承担真实结构：表头、记录边界、时间段和详情章节。
- 编号仅用于真实记录号、排名、步骤或版本，不作为无意义装饰。
- 禁止阴影制造层级；层级通过留白、字体和线条完成。

### 12.6 动效

- 首屏只进行一次 300–500ms 的文字进入编排。
- 列表更新允许数字短暂高亮，随后恢复。
- 链接使用下划线位移或颜色变化反馈。
- 页面切换不使用大幅遮罩或长时间过渡。
- 禁止自动轮播、视差、粒子、3D、持续闪烁和无意义跑马灯。
- `prefers-reduced-motion: reduce` 下关闭所有非必要动画。

### 12.7 可访问性

- 普通文字与背景对比度达到 WCAG AA；核心操作达到 AAA 优先。
- 所有交互可通过键盘完成，焦点样式使用清晰的 2px 实线。
- 不以颜色作为唯一状态表达，酸绿状态必须同时配文字。
- 表格在移动端优先转为可读记录行，不能只依赖横向滚动。
- 二维码必须有文字替代方案。

## 15. 系统架构

### 15.1 总体结构

```text
                     境外采集区
      白名单网页 / RSS / GitHub API / 播客元数据
                         │
                  Collector Containers
                         │ 仅公开内容与元数据
                         │ 加密传输
                         ▼
                     中国大陆区
┌──────────────────────────────────────────────────────┐
│ Ingest API → Object Storage → Queue                  │
│                              │                       │
│                              ├→ Normalize / Dedup    │
│                              ├→ Entity / Cluster     │
│                              ├→ LLM Translate        │
│                              └→ LLM Summarize        │
│                                      │               │
│ Next.js Public Site ← PostgreSQL ← Publish Service  │
│          │                   │                       │
│          └──────── Admin ────┘                       │
└──────────────────────────────────────────────────────┘
```

### 15.2 技术底座

- 前台与后台：Next.js + TypeScript。
- 样式：CSS Variables + 项目内组件层；可使用 Tailwind，但视觉 token 必须集中定义，不能依赖随机 utility 组合形成设计。
- 数据库：PostgreSQL。
- 队列与缓存：Redis。
- 原始内容：S3 兼容的大陆对象存储。
- 数据访问：类型安全 ORM，迁移文件纳入版本控制。
- 采集与处理：独立 Docker 容器；允许按现有开源项目语言封装，不强制全部重写。
- 部署：Docker，云厂商无关；公开站、数据库、后台和境内 LLM 处理位于大陆。
- 统计：自托管、无 Cookie 的轻量分析。
- LLM：统一 Provider Adapter，业务代码不绑定单一厂商。

### 15.3 队列任务

- `source.fetch`
- `raw.normalize`
- `raw.deduplicate`
- `entity.extract`
- `event.cluster`
- `content.translate`
- `content.summarize`
- `event.publish`
- `github.snapshot`
- `frontier.verify`
- `frontier.settle`

任务必须幂等，支持重试与死信状态。模型返回必须经过结构化 Schema 校验，不接受自由文本直接写入数据库。

### 15.4 缓存与更新

- 首页和公开列表采用服务端渲染或增量静态更新。
- 信息流新事件发布后主动失效相关页面缓存。
- GitHub 趋势榜按观察窗口更新，不要求实时秒级变化。
- 边境排行榜至少每日更新；临近结算时可提高频率。

## 16. 安全、合规与内容风险

### 16.1 跨境边界

- 境外节点只采集并向境内传输公开网页、公开仓库、公开 RSS 和公开元数据。
- 用户邮箱、后台会话、运营日志和 OPC 咨询数据不得传至境外节点。
- 境外节点不持有境内后台密码、数据库凭证或 LLM 生产密钥。

### 16.2 采集安全

- 只允许白名单域名和协议，防止 SSRF。
- HTML 清洗后再存储和处理；不执行脚本、宏、仓库代码或 README 指令。
- 对内容长度、文件类型、重定向次数和下载体积设置上限。
- 所有外部文本都按不可信数据处理，不能覆盖系统提示或调用工具。

### 16.3 GitHub 安全

- 只接受标准 `github.com/{owner}/{repo}` 公开仓库地址。
- 不要求用户提供 GitHub Token、密码或 OAuth 授权。
- 挑战码使用高熵随机值，服务端只保存哈希，并与赛季及仓库绑定。
- 验证完成或过期后挑战码不可再次使用。
- 对提交和验证接口设置 IP 与仓库维度的速率限制。

### 16.4 隐私

- 报名邮箱在数据库中加密保存，排行榜和公开接口永不返回。
- 无 Cookie 统计不创建跨站用户画像。
- 纠错表单与报名页明确说明收集目的、保存期限和删除渠道。
- 正式上线前完成隐私政策、用户条款、赛事规则、备案信息和专业服务责任边界复核。

### 16.5 版权与来源

- 公开页面发布摘要、必要短引文和原文链接，不复制完整文章。
- README 快照主要用于内部分析，公开页只展示摘要。
- 来源删除或更正内容后，系统应允许同步下架、更新或标记争议。

- 方法说明、隐私、条款和纠错页面骨架。

验收：

- 首次访问者能在 15 秒内说清产品定位和四个频道关系。
- 任何核心内容可在三次操作内到达。
- 全站无装饰图像、图标、渐变、阴影和大圆角卡片。
- 键盘导航、焦点、对比度和减少动态效果可用。
- 360px、768px、1280px 和 1440px 宽度下无破版。
- 生产构建通过，无明显控制台错误。

### 21.2 第二阶段：平台能力

交付：

- PostgreSQL 数据模型与迁移。
- 共享密码运营后台。
- OPC 菜单管理。
- GitHub 仓库检查、挑战生成、验证和基线记录。
- 排行快照、赛季结算与历史获奖限制。
- 自托管无 Cookie 统计。

验收：

- 未登录用户无法访问后台数据。
- 联系邮箱不出现在公开响应、日志或分析事件中。
- 同一挑战不可跨仓库、跨赛季或重复使用。
- 已获奖仓库无法进入新赛季。
- 排行结果可由基线和最终快照复算。

### 21.3 第三阶段：自动化内容

交付：

- 境外白名单采集节点。
- 境内接收、对象存储、队列和处理器。
- 去重、实体抽取、事件聚合、翻译、摘要与自动发布。
- GitHub 趋势快照与 SiC 自动摘要。
- 失败重试、死信状态、来源暂停和内容下架。

验收：

- 新白名单来源必须人工批准一次才能发布。
- 同一原始内容重复采集不产生重复事件。
- 模型输出不符合 Schema 时不得发布。
- 原始来源、生成时间和 AI 标识在公开页面可见。
- 关闭某来源后不再产生新的公开内容。
- README 中的提示词或脚本不能改变处理流程或触发执行。

## 23. 设计决策摘要

| 决策 | 结论 |
| --- | --- |
| 产品核心 | 一人公司操作系统 |
| 首发人群 | 已商业化或即将注册主体的技术型超级个体 |
| 主要市场 | 中国大陆 |
| 主商业入口 | OPC 固定价格标准服务 |
| OPC 交易 | 网站展示，扫码后线下完成 |
| 信息流 | 白名单采集、境内 LLM、自动发布 |
| 内容单位 | 可持续更新的事件卡 |
| SiC | 趋势 + 自动项目摘要 |
| 趋势排序 | 近期 Star 增速优先 |
| 边境排名 | 报名通过后净新增 Star |
| 赛季 | 自然季度，季度最后一天结算 |
| 重复参赛 | 未获奖仓库可以；获奖仓库不可以 |
| 普通用户账户 | 不建设 |
| 内部后台 | 共享密码、极简运营 |
| 视觉 | 纯文字、黑白灰、低频酸绿 |
| 核心指标 | 每月有效 OPC 咨询数 |
| 部署 | 中国大陆、自托管、云厂商无关 |
