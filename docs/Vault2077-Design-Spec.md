## 1. 文档目的

本文档定义 Vault2077 首发版本的产品边界、信息架构、页面体验、内容模型、视觉系统、数据与系统架构、风控要求、开发阶段和验收标准。它是设计与工程实现的共同基线。

文档权威层级、专项规格和变更规则见[设计文档地图](README.md)。各频道的专项规格在所属频道内优先于本文摘要；研究资料不具有决策权。

首发开发按三个阶段推进：

1. 高保真公开站：完成品牌首页、四个频道、响应式页面、真实路由和示例数据。
2. 平台能力：接入数据库、轻量运营后台、GitHub 仓库验证与排行榜。
3. 自动化内容：部署境外采集、境内 LLM 翻译摘要、事件聚合与自动发布管线。

### 当前实现状态（2026-07-23）

- 第一阶段公开站已完成。
- 边境计划的无账号报名、挑战文件验证、Star 基线和公开排行榜已有可演示实现；“四无”页面、宣言与规则弹窗、奖品捐献、每小时快照、季度自动结算、随机奖池和往届记录仍按专项规格继续完成。

- 首批来源注册表、在线核验和运行 bundle 已实现；境外作业使用 Python 采集 Vault 信息流、使用 Node 22 采集 SiC 固定源。GitHub Actions 四次调度、Vault 签名入站队列、标准 OpenAI 兼容 LLM 处理、事件编排与自动发布已经完成本地全链路验收。生产 LLM 参数按要求留空，仍待部署环境配置；来源组合后台和真实 OPC 联系方式后续接入。

## 2. 产品定义

### 2.1 一句话定位

Vault2077 是面向技术型超级个体的一人公司操作系统，以持续情报为入口，以标准化专业服务为商业核心，以硅碳共生的技术趋势和永续式开放竞赛构建长期社区。

### 2.2 首发用户

核心用户是已经产生收入或即将注册经营主体的技术型超级个体，通常经营 AI 产品、开发者工具、内容/IP 或专业服务，团队规模为 1–3 人。

纯学习者、传统小微企业和普通自由职业者可以使用公开内容，但不主导首发产品设计。

### 2.3 产品层级

| 层级 | 频道 | 核心任务 | 对业务的作用 |
| --- | --- | --- | --- |
| 入口 | Vault 信息流 | 压缩高价值 AI 信息 | 获客与持续回访 |
| 商业核心 | OPC 服务台 | 展示可直接购买的标准服务 | 产生有效咨询与收入 |
| 能力层 | SiC 学院 | 发现正在加速的技术能力 | 建立专业认知与工具心智 |
| 社区层 | 边境计划 | 让公开项目持续参与竞赛 | 社区增长与品牌事件 |

### 2.4 首发商业边界

- Vault 信息流、SiC 学院和边境计划免费、无广告、无会员墙。
- OPC 服务台是网站唯一直接商业入口。
- 普通用户全站无需注册或登录。
- OPC 首发只做服务展示：用户扫码添加工作人员，在线下付款并接受服务。
- 内部运营后台使用共享密码，不建设账号、角色或组织权限系统。
- 核心成功指标为“每月由网站带来的有效 OPC 咨询数”。

### 2.5 明确不做

- 不做面向普通用户的账户、收藏、评论、私信或社区关系链。
- 不做在线支付、订单、材料上传、服务进度或交付物系统。
- 不做传统视频课程、训练营或重型学习管理系统。
- 不做开放网络的无边界抓取。
- 不做资讯全文搬运或付费内容绕过。
- 不做赞助置顶、付费榜单或人工修改竞赛排名。
- 不做图标、插画、装饰性照片、3D 或粒子视觉。唯一长期例外是 OPC 游骑兵协会的专家头像：头像用于识别专家和区分档案，不作为装饰背景；正式档案应与经本人确认的公开职业身份绑定，明确标注为预览的档案可使用预览头像，保持灰阶、直角、无美化滤镜，不得扩展为其他频道的配图许可。

## 4. 设计原则

### 4.1 参考边界

视觉节奏参考 [Xiaomi MiMo](https://mimo.xiaomi.com/) 的单一品牌叙事、巨型文字首屏、克制留白和清晰分区，但不复制其内容结构、品牌元素或具体动效。Vault2077 的频道更复杂、数据更密集，因此首页只借鉴其“先建立主张，再展开内容”的叙事方式；频道页使用适合持续更新记录的高密度列表。

### 4.2 未来档案库，而非赛博朋克海报

视觉由文字、时间、来源、编号、状态和秩序构成。所谓“2077”体现在系统像一套来自未来的可靠基础设施，而不是霓虹城市、机器人、锁孔或终端装饰。

### 4.3 文字账本

全站的标志性设计语言是一套真实、持续更新的文字账本：

- 内容拥有稳定记录号，例如 `VLT/EVT/2026/00731`。
- 项目拥有趋势观察窗口，例如 `24H +184 / 7D +922`。
- 赛事拥有明确状态，例如 `SUMMER 2026 / OPEN / 71 DAYS LEFT`。
- 服务拥有固定范围版本，例如 `OPC/TAX/001 / REV.03`。
- 页面顶部状态行展示真实的更新时间、来源数量或赛季状态，不使用伪数据装饰。

这套账本既承担导航和可信度，也构成 Vault2077 的独特视觉签名。

### 4.4 信息压缩优先

首页不做资讯瀑布；事件不按来源机械复制；卡片不重复标题、摘要和标签。每个界面元素只完成一个任务。

### 4.5 透明而非假装客观

AI 摘要必须标识，排名方法必须解释，原始来源必须可达，事件更新时间必须可见，竞赛规则必须可复核。

### 4.6 极简必须精确

没有图像后，字号、行宽、基线、间距和交互反馈必须严格统一。不能用大面积空白掩盖信息结构；首页采用非对称实时版面，而不是等权卡片或连续长章节。

### 6.1 首页单一任务

让首次访问者在 15 秒内理解 Vault2077 服务谁、提供什么，并能进入最相关的频道；首页是四个频道的总控台，不等同于信息流，也不让任何单一频道垄断首页叙事。

### 6.2 页面结构

### 6.3 首屏行为

- “前方高能！”使用小号中文，“Vault2077”使用单行中等尺度品牌排版；标题区压缩为紧凑横向页头，不设置档案侧栏或装饰性竖排文字。
- 全站不设置独立顶部状态带；更新时间、来源数、赛季状态等元数据并入对应页面的标题区。
- 主体接近全宽，桌面端信息流与右侧栏目约为 `62 : 38`；桌面主体不超过 1.5 屏，移动端不超过 3 屏。

### 6.4 首页频道预览

- 信息流：左侧贯通上下，根据可用空间展示最近 3–4 个事件；首版以 3 个为基准。每条只显示标题与一句话判断，不混入原子资讯。
- SiC：位于右上，展示 3 个趋势项目、中文说明、能力分类和 24 小时 Star 增长。
- OPC：位于右下，只展示 2 项代表性服务、固定价格、周期和一行适用场景，不在首页放二维码或促销文案。
- 边境计划：位于主体下方，以全宽深色招募横幅展示“跨越边境，荒野无垠。”、“无期限 · 无评审 · 无组织 · 无目标”和唯一主操作“参与计划”。

栏目主要依靠灰阶底色、留白和排版建立边界，不使用圆角、投影或装饰性线框。所有呈现为横条的反色反馈必须占满其完整结构区域，不得停在文字或内部内容宽度：单栏横条从浏览器左缘贯通至右缘，双栏横条从对应浏览器外缘贯通至中间分割线，移动端恢复为左右双向贯通。内容条目悬浮或键盘聚焦时标题轻微右移并放大。移动端按信息流、SiC、OPC、边境计划顺序单列展示，并将 SiC 预览压缩为 2 项。

## 7. Vault 信息流

Vault 信息流不是新闻数据库，而是由 AI 持续编排的“事件簿—资讯瀑布”。资讯瀑布承接经翻译与单条摘要处理的境外原子资讯；事件簿沉淀由多条方向一致、相互独立的资讯共同支撑的意义单元。

首版在同一页面左侧展示事件簿、右侧展示资讯瀑布，不使用标签页切换。事件簿每批 10 个事件，资讯瀑布每批约 15 条；只有归入事件的资讯才获得长期留痕价值。删除搜索、标签和筛选能力。事件列表只显示标题和一句话判断，并以带日期的粗线划分每日事件；点击后查看 300–500 字综合摘要、核心实体、完整资讯集合、编号引用和最后更新时间。资讯瀑布只显示中文处理标题与原文标题；资讯详情保留中文处理结果、原始语言内容或合规摘录及原始来源。

事件标题上限为 28 个全角字符，资讯中文标题上限为 36 个；事件判断和资讯摘要均为 20 个全角字符以内的单行文字。所有公开时间统一为北京时间。

完整的领域模型、页面字段、双语规则、批次时间、生命周期、纠错边界和验收标准以 [Vault2077 信息流首版设计基线](Vault2077-Feed-Design-Spec.md) 为准。海外信源范围和采集连接器必须另行讨论，不在本规格中预设结论；人工纠错的权限边界以后台专项规格为准。

## 8. OPC 服务台

完整规则以 [OPC 服务台首版设计基线](Vault2077-OPC-Design-Spec.md) 为准。

### 8.1 交付模式

Vault2077 是服务的直接交付方。内部律师、税务师和其他专业人员将基础服务封装成范围明确、固定价格、固定材料清单、固定交付周期和固定成果的服务项目。

首发网站不收款、不建订单。用户浏览服务后扫描统一工作人员二维码，由工作人员在线下确认适用性、收款并交付。

### 8.2 服务分类

- 法务
- 税务与财务
- 知识产权
- 申报与备案
- 人力资源
- 传媒与传播

### 8.3 标准服务模板

每个正式服务必须具备：

| 字段 | 说明 |
| --- | --- |
| 服务名称 | 使用用户能识别的结果命名 |
| 服务编号 | 例如 `OPC/LEGAL/001` |
| 固定价格 | 明确币种及是否含税 |
| 适用对象 | 哪类一人公司适合购买 |
| 包含事项 | 本价格覆盖的工作范围 |
| 不包含事项 | 触发另行报价的边界 |
| 所需材料 | 用户开始前需要准备的内容 |
| 标准周期 | 从材料齐备后的预计工作日 |
| 交付成果 | 用户最终获得什么 |
| 专业复核 | 律师、税务师等负责人类型 |
| 版本与日期 | 服务范围最后更新时间 |

### 8.4 首发占位服务

以下内容只用于第一阶段界面与信息密度验证，正式上线前必须由专业人员替换并确认：

| 分类 | 示例服务 | 示例价格 | 示例周期 |
| --- | --- | ---: | --- |
| 法务 | 单份标准商业合同审阅 | ¥699 | 3 个工作日 |
| 法务 | 网站用户协议与隐私政策基础包 | ¥1,999 | 7 个工作日 |
| 税务 | 个体经营主体税务健康检查 | ¥899 | 5 个工作日 |
| 税务 | 一人公司基础财税咨询包 | ¥599 | 60 分钟 |
| 知产 | 单类别商标注册申请 | ¥799 起 | 以官方流程为准 |
| 知产 | 软件著作权材料整理与申报 | ¥1,499 | 以官方流程为准 |
| 备案 | ICP 备案材料预审 | ¥499 | 3 个工作日 |
| 人力 | 首位员工入职文件包 | ¥999 | 5 个工作日 |
| 传播 | 创始人基础媒体资料包 | ¥1,999 | 7 个工作日 |

这些价格不得被视为最终承诺。开发环境与设计评审页面需显示“示例内容”状态；生产发布检查必须阻止带有该状态的服务上线。

### 8.5 页面结构

- OPC 首页：服务承诺、分类索引、服务表格、交付方式、统一二维码区、常见问题。
- 服务详情：固定价格、适用/不适用、包含/不包含、材料、周期、成果、二维码操作。
- 二维码旁必须同时提供可复制的工作人员账号文字，避免桌面与无摄像头场景受阻。
- 二维码是业务材料，不属于装饰性图像；未提供正式二维码前使用纯文字占位，不生成虚假二维码。

## 9. SiC 学院

完整规则以 [SiC 学院首版设计基线](Vault2077-SiC-Design-Spec.md) 为准。

### 9.1 首发定位

SiC 学院是固定来源驱动的技术趋势与正式内容阅读区，不是新闻事件流，也不按单篇内容临时筛选。它包含五种平台指标、Skill/MCP 扩展生态榜，以及论文、档案、课程、播客四个内容组。

页面保留全站统一的频道标语区，并在标语区与正文之间设置满宽黑底白字的静态文字缎带：`SILICON × CARBON: WE WILL REDEFINE EVOLUTION.`。正文采用左宽右窄双栏：左侧按“论文、档案、课程、播客”纵向形成杂志式主阅读流；右侧纵向展示四张视觉榜单，其中 GitHub 24H 与 7D 指标共享无外框文字标签切换。左栏反色延伸至浏览器左缘和中线，右栏反色延伸至中线和浏览器右缘。

左侧每个固定来源只显示它的最新更新。目录态使用中文大标题和英文小标题，不展示说明或摘要；点击后才展开一句话说明、内容摘要和直达原文链接。榜单使用呼吸式三角控制 Top 5/Top 10，并以轻量下拉动画显示新增条目。

### 9.2 数据范围

- GitHub Trending、全站 24H 新增 Star、全站 7D 新增 Star。
- Hugging Face 模型近 7 日下载增长与 OpenRouter 官方周使用排序。
- 经来源目录批准的论文追踪、官方技术档案、系统课程和权威主理人播客。

README 始终被视为不可信输入。系统必须剥离脚本和隐藏内容，不执行其中任何指令、链接、代码或安装步骤。

### 9.3 趋势视图

- GitHub Trending 保留官方榜单顺序并显示累计 Star。
- GitHub 增长榜使用标签切换 24H 与 7D 新增 Star，两个时间窗独立计算，不合成趋势分。
- Hugging Face 与 OpenRouter 保留各自已声明的平台口径，不将其解释为全行业市占率或模型能力。

### 9.4 项目列表字段

- 正面只显示仓库或模型名称及当前榜单指标。
- 默认 Top 5，可独立展开至 Top 10。
- 展开控制使用无文字的慢速呼吸三角，第 6–10 条通过下拉动画出现。
- 点击后原地翻转，背面只显示原始地址与复制按钮。
- 页面不展示交叉入榜标记、采集方式或数据处理说明。
- Skill/MCP 扩展生态不使用平台榜的 Top 5/10：折叠 Top 10、展开 Top 20，并分别切换“优选/飙升”。

### 9.5 项目详情

- 原始仓库地址。
- 少于 200 字的中文介绍与应用场景说明。
- 仓库许可证。

详情不复制完整 README，不增加趋势评分、语言、最后提交时间或抓取过程说明。

## 10. 边境计划

完整规则以 [边境计划首版设计基线](Vault2077-Frontier-Design-Spec.md) 为准。

### 10.1 核心规则

- 全程无需用户账号。
- 页面核心原则固定为“无期限、无评审、无组织、无目标”。
- 参赛项目必须是公开、非纯 Fork、未归档且具有可识别开源许可证的 GitHub 仓库。
- 全赛季开放报名，验证通过时记录 Star 基线。
- 赛季结束按净新增 Star 排名。
- 每季第一名成为季度冠军并获得官方冠军奖励。
- 所有最终资格有效的报名按最终排名顺序，从匿名随机奖池中各随机获得至多一件奖品。
- 非冠军仓库可在后续赛季重新报名；季度冠军仓库永久退出后续比赛。
- 网站不以项目类型、完成度、价值、README 质量或主动传播行为进行主观评审。

### 10.2 赛季周期

赛季按自然季度连续运行：

| 赛季 | 结算日 |
| --- | --- |
| 春季 | 3 月 31 日 |
| 夏季 | 6 月 30 日 |
| 秋季 | 9 月 30 日 |
| 冬季 | 12 月 31 日 |

结算时间统一为北京时间 23:59:59。新赛季在下一自然日 00:00:00 开始。

### 10.3 无账号报名流程

```text
输入 GitHub 仓库地址、私人联系邮箱和一句话项目说明
    ↓
查看规则弹窗并勾选同意
    ↓
检查公开、非纯 Fork、未归档、许可证和历史冠军状态
    ↓
生成与“仓库 + 赛季”绑定的一次性挑战码
    ↓
用户提交 .vault2077/season-YYYY-QN.json
    ↓
用户返回页面点击“验证仓库”
    ↓
系统读取默认分支文件并匹配挑战码
    ↓
记录当时 Star 基线并加入排行榜
```

验证文件建议格式：

```json
{
  "platform": "vault2077",
  "season": "2026-Q3",
  "repository": "owner/repository",
  "challenge": "一次性随机字符串",
  "issuedAt": "2026-07-21T12:00:00+08:00"
}
```

挑战码不是用户密钥，不具备其他权限。它只能验证提交者可以修改指定仓库的默认分支。挑战文件必须保留到赛季结算；网站在报名和结算时各检查一次。

### 10.4 报名字段

公开字段：仓库地址、项目名称、项目说明、许可证、报名时间、基线 Star、当前 Star、净新增 Star。

私密字段：联系邮箱、挑战码、验证日志、异常复核记录。私密字段只存储在中国大陆服务端，不出现在仓库或排行榜。

### 10.5 排名与结算

```text
net_new_stars = final_star_count - baseline_star_count
```

- 排行榜展示当前 Star、基线 Star 和净新增 Star，保证结果可理解。
- 系统每小时记录一次 Star，页面显示最近更新时间，结算日取得最终快照。
- 结算时重新检查公开状态、Fork、归档、许可证、默认分支和挑战文件；不引入主观项目评分。
- 默认同分规则：累计 Star 较高者优先；仍相同则报名验证时间较早者优先。该规则在首季开始前写入正式赛事条款。

### 10.6 奖池

- 官方冠军奖励与匿名随机奖池分开展示，官方冠军奖励不参与随机抽取。
- 奖品捐献只收集名称、说明和 Email，提交前必须同意奖品捐献须知。
- 捐献先进入待确认状态，确认后才匿名显示在公开奖池；结算时仍未确认的奖品转入下一赛季。
- 网站在结算后按最终排名顺序自动随机分配，每个仓库每赛季最多一件；剩余奖品顺延。
- 随机分配每赛季只能执行一次，结果保存并进入往届记录。
- 首发不建设支付、物流、仓储、站内联系或所有权托管。

### 10.7 主要页面

- 边境计划首页：固定大标语、宣言弹窗入口、四无、当前赛季、实时前五、官方冠军奖励、随机奖池、报名与捐献入口、往届记录。
- 提交页：仓库地址、邮箱、项目说明和隐私提示。
- 验证页：挑战文件内容、文件路径、提交步骤、验证状态。
- 排行榜：排名、项目、基线、当前值、净新增值、报名日期。
- 奖品捐献页：奖品名称、说明、Email、捐献须知和同意确认。
- 规则不设独立页面，由报名入口的可访问弹窗展示。
- 往届结果不设独立页面，在首页底部展示历季冠军和随机奖品分配。

## 11. 内部运营后台

完整规则以 [后台与运营首版设计基线](Vault2077-Admin-Operations-Spec.md) 为准。

### 11.1 登录

- 单一共享高强度密码。
- 密码只保存 Argon2id 哈希。
- 登录接口限速，连续失败触发短期锁定。
- 使用安全、HttpOnly、SameSite Cookie 保存短期会话。
- 后台路径不依赖“隐藏 URL”作为安全措施。
- 记录操作时间、动作和对象，但不承诺识别具体操作者。

### 11.2 功能范围

- 来源组合：批准修订、暂停、恢复、查看已展开通道和最近采集状态。
- 内容异常：下架、恢复、重算摘要、拆分事件、屏蔽原始条目。
- 采集任务：最近运行、失败原因、重试。
- OPC 菜单：服务项目、分类、价格、状态和版本。
- 边境计划：报名复核、取消资格、赛季、奖品和结算状态。
- 系统：共享密码轮换、基础统计和操作日志。

不做富文本页面搭建器、多人审批、角色权限、复杂数据大屏。

## 12. 视觉系统

### 12.1 色彩

| Token | 名称 | 色值 | 用途 |
| --- | --- | --- | --- |
| `--paper` | Register Paper | `#F3F3EF` | 默认浅色背景 |
| `--paper-bright` | Reading White | `#FBFBF8` | 导航、表单与反相文字 |
| `--carbon` | Registry Ink | `#11120F` | 主文字、结构边界 |
| `--graphite` | Graphite | `#3F413D` | 次级深色文字 |
| `--ash` | Archive Ash | `#74766F` | 辅助信息 |
| `--rule` | Filing Rule | `#C9CAC4` | 分隔线与表格线 |
| `--instrument` | Instrument Black | `#181C1E` | OPC 深色区与仪器式数据区 |
全站不设置品牌强调色。实时状态和关键交互通过黑白反相、边框、字重与位置表达，禁止使用荧光色和渐变制造“科技感”。

OPC 游骑兵协会的专家头像是全站纯文字规则的唯一图像例外。头像必须承担身份识别功能，具有明确替代文本或与同一档案中的姓名、职业身份共同出现；列表和详情不得只靠头像传达信息。除该例外及业务所需二维码外，其他频道仍不得增加照片、插画或装饰性图像。

### 12.2 字体

- 中文正文与界面：本地托管的 `Noto Sans SC Variable`，使用 400、500、700 三个层级。
- 中文品牌主张与页面大标题：`Noto Serif SC`，使用 600；只在需要建立制度感和阅读停顿时出现。
- 全站英文、数字与英文项目名：本地托管的 `Manrope Variable`，采用 SIL Open Font License，使用 400、500、700 三个层级。
- 编号、时间、数据和状态沿用 `Manrope Variable`，开启等宽数字特性；不再依赖系统英文回退字体。
- 字体文件必须放在大陆对象存储或随应用发布，不依赖 Google Fonts 等境外运行时资源。

中文主标语不做无意义的超宽字距；英文品牌字标允许紧缩字距。数据表保持等宽数字。

### 12.3 字号建议

| 角色 | 桌面 | 移动 | 行高 |
| --- | ---: | ---: | ---: |
| 品牌字标 | `14–44px` | `14–32px` | `1` |
| 页面主标题 | `58–142px` | `46–74px` | `0.98–1.08` |
| 区块标题 | `52–88px` | `42–58px` | `1.05–1.15` |
| 正文 | `16–18px` | `16px` | `1.65–1.8` |
| 数据/标签 | `11–13px` | `11–12px` | `1.3–1.5` |

### 12.4 网格与间距

- 桌面内容最大宽度：1440px；页面左右安全边距：32–64px。
- 采用 12 列网格，但阅读正文限制在 720–820px。
- 平板切换为 8 列，移动端为 4 列。
- 间距基准为 4px，主要区块使用 48、72、96、144px 节奏。
- 列表优先采用整行记录形式，避免把所有内容装进圆角卡片。
- 默认圆角为 0；表单控件最多 2px，不能使用大圆角胶囊。

### 12.5 分隔与编号

- 1px 分隔线承担常规结构：表头、记录边界和详情章节；信息流事件簿以带日期的 4px 粗线作为明确的每日分割。
- 编号仅用于真实记录号、排名、步骤或版本，不作为无意义装饰。
- 禁止阴影制造层级；层级通过留白、字体和线条完成。

全站签名结构是“修订脊柱”：只呈现真实版本、服务对象、系统范围和运行状态。SiC 与边境计划可在此基础上使用刻度式列线和等宽数据，OPC 通过反相深色区域表达直接交付责任；四个频道共享制度框架，但拥有不同的信息节奏。

### 12.6 动效

- 首屏只进行一次 300–500ms 的文字进入编排。
- 列表更新允许数字短暂高亮，随后恢复。
- 链接使用下划线位移或颜色变化反馈。
- 页面切换不使用大幅遮罩或长时间过渡。
- 禁止自动轮播、视差、粒子、3D、持续闪烁和无意义跑马灯。
- `prefers-reduced-motion: reduce` 下关闭所有非必要动画。

### 12.7 可访问性

- 普通文字与背景对比度达到 WCAG AA；核心操作达到 AAA 优先。
- 所有交互可通过键盘完成，焦点样式使用清晰的 2px 实线。
- 不以颜色作为唯一状态表达，所有状态必须同时配有明确文字。
- 表格在移动端优先转为可读记录行，不能只依赖横向滚动。
- 二维码必须有文字替代方案。

## 13. 内容与界面文案

### 13.1 语气

- 使用主动语态、短句和具体动词。
- 不使用“赋能、生态、颠覆、无限可能”等空泛词。
- 不用末日、生存或地下黑客叙事包装正常服务。
- 错误信息说明发生了什么以及如何继续，不说“系统开小差了”。

### 13.2 操作命名

| 场景 | 使用 | 不使用 |
| --- | --- | --- |
| 阅读信息 | 查看事件 | 探索更多 |
| 打开来源 | 阅读原文 | 立即前往 |
| OPC | 查看服务范围 | 解锁服务 |
| 赛事入口 | 参与计划 | 加入冒险 |
| 仓库验证 | 验证仓库 | 激活密钥 |
| 错误重试 | 重新验证 | 再试试看 |

### 13.3 空状态

- 信息流无数据：“今日尚无新记录。系统会在发现可信更新后自动发布。”
- 趋势榜无数据：“当前观察窗口还没有足够数据，下一次快照后更新。”
- 赛事无报名：“本赛季尚无通过验证的项目。提交第一个公开仓库。”
- OPC 无正式菜单：“服务范围正在由专业团队确认，暂不接受咨询。”

### 15.2 技术底座

以下是生产目标架构，不应被误读为当前已经部署。当前 MVP 使用一个 Next.js Node.js 22 单写实例和 `VAULT2077_DATA_DIR` 下的原子 JSON 文件；Vault 队列与 SiC 四个数据文件都不支持多副本并发写入。境外 GitHub Actions 同时安装 Python 3.12 与 Node.js 22，无 Docker、数据库、浏览器或 LLM。迁移到 PostgreSQL/Redis/对象存储前，以 [配置与部署手册](Vault2077-Deployment-Configuration-Manual.md) 的单实例拓扑为唯一运行依据。

- 前台与后台：Next.js + TypeScript。
- 样式：CSS Variables + 项目内组件层；可使用 Tailwind，但视觉 token 必须集中定义，不能依赖随机 utility 组合形成设计。
- 数据库：PostgreSQL。
- 队列与缓存：Redis。
- 原始内容：S3 兼容的大陆对象存储。
- 数据访问：类型安全 ORM，迁移文件纳入版本控制。
- 采集与处理：独立 Docker 容器；允许按现有开源项目语言封装，不强制全部重写。
- 部署：Docker，云厂商无关；公开站、数据库、后台和境内 LLM 处理位于大陆。
- 统计：自托管、无 Cookie 的轻量分析。
- LLM：统一 Provider Adapter，业务代码不绑定单一厂商。

### 15.4 缓存与更新

- 首页和公开列表采用服务端渲染或增量静态更新。
- 信息流新事件发布后主动失效相关页面缓存。
- GitHub 趋势榜按观察窗口更新，不要求实时秒级变化。
- 边境排行榜至少每日更新；临近结算时可提高频率。

## 16. 安全、合规与内容风险

### 16.1 跨境边界

- 境外节点只采集并向境内传输公开网页、公开仓库、公开 RSS 和公开元数据。
- 用户邮箱、后台会话、运营日志和 OPC 咨询数据不得传至境外节点。
- 境外节点不持有境内后台密码、数据库凭证或 LLM 生产密钥。

### 16.2 采集安全

- 只允许来源注册表中的域名和协议，防止 SSRF。
- HTML 清洗后再存储和处理；不执行脚本、宏、仓库代码或 README 指令。
- 对内容长度、文件类型、重定向次数和下载体积设置上限。
- 所有外部文本都按不可信数据处理，不能覆盖系统提示或调用工具。

### 16.3 GitHub 安全

- 只接受标准 `github.com/{owner}/{repo}` 公开仓库地址。
- 不要求用户提供 GitHub Token、密码或 OAuth 授权。
- 挑战码使用高熵随机值，服务端只保存哈希，并与赛季及仓库绑定。
- 验证完成或过期后挑战码不可再次使用。
- 对提交和验证接口设置 IP 与仓库维度的速率限制。

### 16.4 隐私

- 报名邮箱在数据库中加密保存，排行榜和公开接口永不返回。
- 无 Cookie 统计不创建跨站用户画像。
- 纠错表单与报名页明确说明收集目的、保存期限和删除渠道。
- 正式上线前完成隐私政策、用户条款、赛事规则、备案信息和专业服务责任边界复核。

### 16.5 版权与来源

- 公开页面发布摘要、必要短引文和原文链接，不复制完整文章。
- README 快照主要用于内部分析，公开页只展示摘要。
- 来源删除或更正内容后，系统应允许同步下架、更新或标记争议。

## 17. 数据分析

### 17.1 核心指标

有效咨询的首发代理定义：用户查看服务列表或详情后触发二维码查看、复制工作人员账号或确认“已添加工作人员”的事件。运营端可按月人工确认其中的真实需求数。

### 17.2 辅助指标

- 信息流 7 日回访访客比例。
- 事件详情到原文的点击率。
- OPC 列表到详情的进入率。
- OPC 详情到联系方式的转化率。
- SiC 项目到 GitHub 的跳转率。
- 边境计划提交开始、挑战生成、验证成功的漏斗。
- 每赛季有效参赛仓库数。

## 18. SEO 与分享

- 所有公开事件、服务、项目和赛季拥有可索引的稳定 URL。
- 服务端输出标题、描述、canonical、结构化数据和站点地图。
- 事件页使用来源与更新时间生成可信的搜索摘要。
- 纯文字视觉不依赖分享图片表达含义；首发分享卡可使用服务端生成的文字排版图，但不作为网站页面图像。如果严格执行“任何图像均不生成”，则使用平台默认链接预览。
- 管理后台、验证中页面和私密状态禁止索引。

## 19. 响应式行为

### 19.1 桌面

- 12 列网格显示完整来源、时间和趋势数据。
- 悬停只增强可点击文字，不移动整行造成阅读跳动。
- 详情页正文与右侧元数据可形成 8+4 布局。

### 19.2 平板

- 8 列网格；状态信息允许换行。
- OPC 列表保留价格和周期，次要字段进入详情。

### 19.3 移动

- 品牌字标完整换行，不横向裁切正文。
- 列表转换为上下排列的记录行，不使用小字号压缩桌面表格。
- 顶部导航通过文字“菜单”展开。
- 固定底部按钮只允许出现在 OPC 服务详情和边境提交关键步骤，且必须是文字按钮。

## 20. 第一阶段示例内容要求

高保真公开站必须使用足以验证信息密度的中文示例内容：

- 8–12 个 Vault 事件，覆盖公司公告、人物观点、播客和文章。
- 9 个 OPC 示例服务，全部带“示例内容”状态。
- 12–20 个 SiC 项目，覆盖不同增长速度和项目类别。

## 21. 开发阶段与验收标准

### 21.1 第一阶段：高保真公开站

交付：

- 首页和四个频道的列表、详情与关键交互页面。
- 响应式桌面、平板和移动布局。
- 本地结构化示例数据与真实路由。
- OPC 二维码文字占位。
- 边境计划提交和验证的完整前端状态演示。
- 方法说明、隐私、条款和纠错页面骨架。

验收：

- 首次访问者能在 15 秒内说清产品定位和四个频道关系。
- 任何核心内容可在三次操作内到达。
- 全站无装饰图像、图标、渐变、阴影和大圆角卡片。
- 键盘导航、焦点、对比度和减少动态效果可用。
- 360px、768px、1280px 和 1440px 宽度下无破版。
- 生产构建通过，无明显控制台错误。

### 21.2 第二阶段：平台能力

交付：

- PostgreSQL 数据模型与迁移。
- 共享密码运营后台。
- OPC 菜单管理。
- GitHub 仓库检查、挑战生成、验证和基线记录。
- 每小时排行快照、赛季自动结算、季度冠军限制、奖品捐献和一次性随机分配。
- 自托管无 Cookie 统计。

验收：

- 未登录用户无法访问后台数据。
- 联系邮箱不出现在公开响应、日志或分析事件中。
- 同一挑战不可跨仓库、跨赛季或重复使用。
- 季度冠军仓库无法进入新赛季，随机奖品获得仓库仍可重新报名。
- 排行结果可由基线和最终快照复算。

### 21.3 第三阶段：自动化内容

交付：

- 境外受控来源采集节点。
- 境内接收、对象存储、队列和处理器。
- 去重、实体抽取、事件聚合、翻译、摘要与自动发布。
- GitHub 趋势快照与 SiC 自动摘要。
- 失败重试、死信状态、来源暂停和内容下架。

验收：

- 新来源组合修订必须人工批准一次；获批组合中明确标注原始来源的通道自动进入来源注册表。
- 同一原始内容重复采集不产生重复事件。
- 模型输出不符合 Schema 时不得发布。
- 原始来源、生成时间和 AI 标识在公开页面可见。
- 关闭某来源后不再产生新的公开内容。
- README 中的提示词或脚本不能改变处理流程或触发执行。

## 23. 设计决策摘要
