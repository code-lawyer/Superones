# Vault2077 SiC 学院：内容信息架构规划

> 本文是内容架构决策记录，**不作为来源准入或页面布局的最终依据**。当前规范以 [SiC 学院设计基线](Vault2077-SiC-Design-Spec.md) 为准；已批准的完整目录、固定入口和采用理由以 [SiC 首批固定信息源目录](Vault2077-SiC-Source-Catalog.md) 为准。

> 状态：核心内容分组已确认并实现；早期候选判断保留用于追溯。
> 当前公开结构：五项平台指标、Skill/MCP 扩展生态，以及四个固定来源内容组。

## 结论

SiC 不是新闻页，也不应该试图收录“所有重要技术动态”。它应是一个由严格准入、持续维护的技术发布源构成的技术趋势观察与学习空间：**先审核整条来源，再完整接入该来源的正式技术更新；不对单条内容二次筛选。**

五个排行榜作为独立的**指标榜**层保留；其下建议设置 **4 个内容组**。这里用“内容组”而不用“栏目/频道”，以免与 Vault2077 的一级频道概念混淆。

| 层级 | 组成 | 要回答的问题 |
| --- | --- | --- |
| 指标榜 | GitHub Trending、GitHub 24H 新增星标、GitHub 7D 新增星标、Hugging Face 近 7 日模型下载增长、OpenRouter 周模型使用 | 此刻哪些开源项目和模型正在升温？ |
| 内容组 | 论文、档案、课程、播客 | 值得系统阅读、观看、追踪的技术内容有哪些？ |
| 工具能力 | 论文/作者/引用检索、论文解析、论文—代码关联 | 打开内容后如何继续研究？ |

指标榜不与论文、视频、播客混排；它们只描述各自平台和窗口内的活动，不承担质量、安全、能力或投资评级。

## 四个内容组

| 内容组 | 解决的任务 | 接纳的正式发布事件 | 首批来源候选 | 不纳入 |
| --- | --- | --- | --- | --- |
| **论文** | 高密度发现近期值得读的 AI / 硬核技术论文 | 被既定论文追踪源正式收录或推荐的论文、技术报告 | Hugging Face Daily Papers；DAIR Papers of the Week | 把 arXiv 全量投稿直接变成信息流；社交媒体“论文速报” |
| **档案** | 保存并追踪技术机构的正式公开表达，让用户同时看到研究、工程能力、产业展望与人与 AI 关系的思考 | 官方研究发布、技术报告、系统卡、开发者博客、Changelog、Release Notes、产品/行业展望、组织正式新闻与政策说明 | OpenAI、Anthropic、Google DeepMind / Google Research、Meta AI、Microsoft Research / Foundry、NVIDIA Research / Developer 等 | 个人技术博客、社交短帖、媒体转述、网页 diff、代码提交、非官方评论 |
| **课程** | 追踪系统课程、讲座、研究演讲与工程演示 | 新课程、课程单元、公开讲座、研究报告视频、技术 workshop | Google ML Courses / Kaggle Learn；Google DeepMind；Microsoft Research；NVIDIA Developer；Hugging Face；后续审核 Berkeley AI、MIT CSAIL 等 | 发布会宣传片、泛 AI 工具教学、非技术内容 |
| **播客** | 通过权威技术人物的长期对话，理解技术、产业、科学与社会影响 | 经审核的权威主理人所正式发布的每一期节目 | 由主持人身份审核后的 Dwarkesh、Lex Fridman、Latent Space、Google DeepMind Podcast、The Cognitive Revolution 等 | 只挑某嘉宾或某主题的单集；仅因流量而准入的泛娱乐节目 |

### 学术领域的两条线

- **论文**保留可信论文追踪源的收录与排序，回答“近期先读什么”。
- **档案**保留技术机构的原始发布节奏，回答“这些机构正式发布了什么研究、能力变化与行业判断”。

二者分开，既不会把全量 arXiv 误当精选内容，也不会把一手研究埋没在外部论文榜中。硬科技（系统、芯片、机器人、科学智能等）通过已准入机构的正式档案覆盖；不要把它伪装成一个没有边界的“全球科技突破”信息流。

### 为什么只保留一个“档案”组

研究发布、开发者文档、产品与产业展望在 OpenAI、Anthropic 等机构的正式站点中本就经常交织。拆成“研究”“官方技术档案”“深度技术解读”会造成重复，也会迫使用户猜测一篇官方文章该归到哪里。因此它们统一进入“档案”，以发布机构自身的正式站点为唯一准入边界。

档案不是狭义 API 文档区：一份模型系统卡、一篇对社会影响的官方研究文章、一项产业判断或组织正式新闻，只要来自已准入技术机构的正式发布源，均可进入。个人技术博客、独立论文讲解、复现文章则全部不进入 SiC。

### 论文工具不应伪装成信息源

Semantic Scholar、Explainpaper 是研究辅助工具；Papers with Code 适合作为论文—代码与基准的关联入口。它们可在论文详情页的“继续探索”中出现，但不应作为 SiC 的资讯流。arXiv 只可作为获选论文补齐元数据的底座，不直接构成主内容源。

## 当前页面组织

早期“三个 GitHub 榜在上、两个模型榜在下、四块内容再往下”的方案已经废止。当前结构遵循“阅读为主、指标为辅”：

1. 全站统一频道标语区之后，是满宽黑底白字缎带 `SILICON × CARBON: WE WILL REDEFINE EVOLUTION.`。
2. 正文为左宽右窄双栏，中间只有一条连续细线。
3. 左栏按“论文、档案、课程、播客”纵向排列。每个来源只显示其最新更新；折叠态只有日期、来源身份、中文大标题和英文小标题，点击后才出现一句话说明、内容摘要和原文链接。
4. 右栏为四张视觉榜：GitHub Trending、GitHub 增长趋势、Hugging Face、OpenRouter。GitHub 24H/7D 是两项独立指标，但在同一视觉榜中以标签切换。
5. Skill/MCP 扩展生态位于平台榜之后，用两层标签切换对象与“优选/飙升”。
6. 平台榜折叠 Top 5、展开 Top 10；Skill/MCP 折叠 Top 10、展开 Top 20。
7. 移动端先显示四个内容组，再显示趋势榜；顶部提供论文、档案、课程、播客、趋势榜的粘性索引。

| 项目 | 当前首页行为 | 进一步动作 |
| --- | --- | --- |
| 四个内容组 | 每来源最新一条，每组最多 6 条 | 原地展开摘要并直达正式原页 |
| 五项平台指标 | 四张视觉榜，Top 5/10 | 榜单行原地翻板，显示地址与复制按钮 |
| Skill/MCP | 两对象 × 两口径，Top 10/20 | 直达正式市场详情与接入入口 |

## 内容卡片与运行规则

内容目录态只提供足以决定“要不要打开”的信息：双语标题、发布时间与来源身份。一句话说明和内容摘要不在目录态直接铺开，而在点击后原地展开。榜单目录只显示名称和一个指标；点击后原地翻板显示地址与复制按钮，不在行下方追加说明。

1. **来源先于内容**：每个发布源单独审核、单独确认后，才完整接入其正式更新。
2. **以正式发布边界同步**：课程、播客、研究发布、changelog 分别按各自天然事件更新，不做网页 diff 采集。
3. **播客按主理人准入**：审核主持人或节目主理人在技术、科学、工程或产业中的权威身份与长期影响力；一经准入，完整接入节目，不按嘉宾或主题筛选单集。
4. **不使用社交观点**：不因作者知名而纳入 X / Twitter、个人短帖或转述。
5. **同一来源只归属一个内容组**：避免内容跨区重复。
6. **原始链接优先**：SiC 建立可靠入口和阅读秩序，用户始终能回到原始发布页。

## 已完成的来源决策

首批来源已经一次性审核完成，共 27 个：论文 2、档案 12、课程 8、播客 5。运行时只读取 `config/sic-source-registry.json` 中的 `approved` 条目，不再逐条向用户确认。新增、暂停或替换来源时，必须先修改并审阅正式来源目录，再改变注册表状态；不能用某一条内容“看起来不错”作为临时接入理由。
